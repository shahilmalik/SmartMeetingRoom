Smart Meeting Room — Hardware & Build Brief

Reference document for continuing this project (Smart Cities and IoT, Summer 2026). Hand this to any AI assistant or collaborator as starting context.

1. Project Brief

A smart meeting room continuously monitors its environment — occupancy, temperature, humidity, CO2, noise, and ambient light — and uses an AI planner to decide how to respond: adjusting ventilation, dimming/brightening lights, switching heating/cooling, and cutting power to unused devices.

  • Goal: when occupied → comfortable and healthy; when empty → energy-saving state.
  • Planning: live sensor readings are turned into a PDDL problem instance and solved by Fast Downward, which returns an ordered action plan executed step by step.
  • Replanning triggers: occupancy change, a sensor crossing a threshold (e.g. CO2 too high, temperature out of comfort range), a periodic 60-second re-evaluation, and a manual override from the dashboard.
  • Communication: fully decoupled via an MQTT broker (Mosquitto) — modules only publish/subscribe to topics, never talk to each other directly.
  • Modules: iot_node (Pi — sensors/actuators), state_manager (digital twin), problem_generator (state+goal → PDDL), planner (Fast Downward wrapper), executor (runs the plan, requests replans), dashboard (live visualisation + manual control).

2. Hardware We Have

  • Raspberry Pi 5 (Model B Rev 1.1), Debian 13 "trixie"
  • GrovePi+ board mounted on the Pi's 40-pin header
    Important build note: The Pi 5's hardware I2C (RP1 controller, /dev/i2c-1) does not tolerate the GrovePi AVR's I2C clock-stretching. We fixed this with a custom software (bit-banged) I2C implementation using lgpio on GPIO2/3 (~/grove_sw_i2c.py), wired into a patched ~/grovepi.py. The GrovePi's onboard ATmega328P firmware was also re-flashed (v1.4.0) via a custom AVR ISP script over GPIO 8/9/10/11, since it shipped with corrupt/missing flash. Any new sensor/actuator code on the Pi must go through this patched grovepi.py (which already provides the standard grovepi API — analogRead, digitalWrite, dht(), etc.) — no further low-level fixes should be needed.
  • Sensors (Grove modules + cables):
    • DHT11 Temperature & Humidity sensor (blue)
    • Light sensor
    • Sound sensor
    • Ultrasonic ranger
    • Button
  • Actuators (Grove modules + cables):
    • Relay module
    • LED
  • Cabling: standard Grove 4-pin cables (white connectors, male header pins on the GrovePi side) — one per module, connecting to the GrovePi's labelled digital (D), analog (A), or I2C sockets.
  • Not available / not used: Aeotec MultiSensor 6 (Z-Wave), Plugwise smart socket (ZigBee), Raspberry Pi Relay Shield, dimmable lamp — all replaced by GrovePi equivalents or simulated (see below).
  • In the kit but not wired in (not needed for required scope): rotary angle sensor, buzzer.

5. Wiring / Port Connections (GrovePi+)

GrovePi+ ports are not interchangeable — the 4-pin Grove cable fits any socket, but each socket is wired to a specific function on the onboard ATmega. D2–D8 are digital ports (D3/D5/D6 also support PWM), A0–A2 are analog ports (for variable-voltage sensors), and I2C-1/2/3 share one I2C bus (not needed for this build).

#ModulePort type neededConnect togrovepi port #
1 DHT11 (temp/humidity) Digital D4 (already connected) 4
2 LED (dim/brighten) Digital + PWM D3 3
3 Relay (fan/heater) Digital D5 5
4 Ultrasonic ranger (occupancy) Digital D7 7
5 Button (manual override) Digital D8 8
6 Light sensor Analog A0 0
7 Sound sensor Analog A1 1

This uses all 7 digital ports and 2 of 3 analog ports. LED is on D3 specifically because it needs PWM for true dimming (only D3/D5/D6 on GrovePi+ support analogWrite()). Light/sound sensors must use A0–A2 since digital ports have no ADC.

6. Summary — What's Real vs. What's Simulated

  • 5 sensors total: 4 real (DHT11, light, sound, ultrasonic+button) + 1 simulated (CO2)
  • 4 required actuators: 2 real (relay, LED) + 2 simulated (AC state machine, blinds)
  • Not wired in: rotary angle sensor, buzzer — not needed for the required scope
  • Optional simulated extra: smart-socket device power control

8. MQTT Topic Map

  • sensors/{id} — readings published by the Pi (temperature, humidity, light, noise, occupancy, co2)
  • actuators/{id}/cmd — commands delivered to the Pi (relay, led)
  • planning/problem, planning/plan — PDDL problem instances and resulting plans
  • state/current — current world state (digital twin) maintained by state_manager
  • events/* — replan triggers (occupancy change, threshold breach, periodic tick, manual override)
